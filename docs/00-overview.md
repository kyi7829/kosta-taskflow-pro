## 읽는 순서

```
00-overview → 01-product → 02-specs → 03-design → 04-tasks → 05-conventions
```

| 단계 | 목적 |
|---|---|
| `00` 읽기 | 전체 지형 파악. 다음에 무엇을 읽어야 할지 결정한다. |
| `01` 읽기 | "무엇을 만드는가"를 이해한다. 기술보다 제품을 먼저 파악한다. |
| `02` 읽기 | "어떻게 만드는가"를 이해한다. 기술 결정의 근거를 확인한다. |
| `03` 읽기 | "어떻게 보이는가"를 이해한다. 구현 전 화면 구조를 머릿속에 그린다. |
| `04` 읽기 | "지금 무엇을 해야 하는가"를 파악한다. 현재 작업 맥락을 잡는다. |
| `05` 읽기 | "어떤 규칙으로 쓰는가"를 내면화한다. 이후 모든 코드 작성의 기준이 된다. |

## 왜 이렇게 나누는가 — 관심사 분리 원칙

문서도 코드와 같은 원칙을 따른다.

코드에서 **관심사 분리(Separation of Concerns)** 란 하나의 모듈이 하나의 책임만 지는 것을 의미한다. 문서도 마찬가지다. 하나의 파일에 제품 정의, 기술 명세, 디자인, 작업 계획이 뒤섞이면 다음 문제가 생긴다:

- **변경 충돌**: 기술 스택이 바뀌면 제품 정의 문서까지 수정해야 한다.
- **탐색 비용**: 특정 정보를 찾으려면 전체 문서를 뒤져야 한다.
- **역할 혼동**: 기획자는 `02-specs.md`를, 개발자는 `01-product.md`를 읽을 이유가 없다.

각 파일의 책임을 명확히 분리하면:

| 원칙 | 적용 |
|---|---|
| 단일 책임 | 파일 하나 = 관심사 하나. `01-product`는 제품만, `02-specs`는 기술만 다룬다. |
| 낮은 결합도 | 디자인이 바뀌어도 `03-design.md`만 수정한다. 다른 파일은 영향받지 않는다. |
| 높은 응집도 | 기술 결정에 관한 모든 내용은 `02-specs.md` 한 곳에 모여 있다. |
| 읽는 사람 중심 | 역할에 따라 필요한 문서만 선택적으로 읽을 수 있다. |
